# SW Contract Scopeguard Orchestrator

SW 외주개발계약에서 소송으로 가기 전 분쟁을 작게 만들기 위한 계약서·부속서류 작성 보조 스킬입니다.

법률 자문, 공증, 판결 예측을 하지 않습니다. 계약 전 과업 범위를 구조화하고, 공급자와 수요자가 합의할 파라미터를 계약 조항과 부속서류에 반영합니다.

이 스킬은 최상단 orchestrator입니다. 특정 문서를 작성할 때는 `references/documents/index.md`에서 대상 module을 찾고, 해당 폴더의 `recipe.md`만 추가로 읽습니다.

## Workflow

1. 작성 관점을 고른다: `발주자`, `수급자`, `중립`. 사용자가 고르지 않으면 `중립`을 기본값으로 둔다.
2. 계약 파라미터 시트를 채운다. 값이 비어 있으면 기본 프리셋을 넣고 `확인 필요`로 표시한다.
3. 기능 ID와 견적 row를 연결한다.
4. 기능별 진입점, 디자인 요건 충족 여부, 디자인 플로우, 최종 화면/결과를 입력한다.
5. 견적 row별 전제조건, 완수 조건, 테스트 조건, 부수 항목을 입력한다.
6. 필수 입력값 누락 여부를 확인한다.
7. 내부 점검 기준을 계약 조항과 부속서류 표에 반영한다.
8. 표 형태 DOCX/XLSX 계약문서 패키지를 만든다.
9. 계약서 초안과 부속서류를 검토한다.
10. 참고문헌은 별도 문서로 분리한다.

## Required Parameter Groups

각 항목은 단일 선택지로 뭉개지 말고, 계약상 의미가 다른 값을 분리한다.

### 기본 정보

- 계약 관점: 발주자 / 수급자 / 중립
- 프로젝트명, 발주자, 수급자
- 계약 유형: 신규 개발 / 추가 개발 / 유지보수 / 디자인 포함 / AI/API 연동
- 계약금액, 착수일, 납품일, 계약 종료일

### 착수자료

각 자료는 `있음 / 없음 / 계약 후 제공 / 기능별 제공 / 마일스톤별 제공`으로 둔다.

- PRD
- 기능정의서
- 화면설계서/Figma
- 사용자 플로우
- 디자인 기준
- 콘텐츠/문구
- 데이터/API 정보
- 검수기준

### 회의/커뮤니케이션

- 온라인 정기회의: 없음 / 주간 / 격주 / 월간 / 마일스톤별
- 대면회의 포함 횟수: 0회 / 1회 / 2회 / 직접 입력
- 대면회의 1회 기준 시간: 60분 / 90분 / 120분
- 추가 대면회의 과금: 없음 / 시간당 / 반일 / 일일
- 회의록 승인 기한: 1영업일 / 2영업일 / 3영업일
- 구두요청 효력: 없음. 플랫폼/이메일/회의록 승인 시에만 효력

### 수정

- 무상수정 방식: 없음 / 전체 총 N회 / 마일스톤별 N회 / 화면 묶음별 N회
- 수정 1회 기준: 통합 수정요청 1세트
- 수정요청 제출 기한: 산출물 제출 후 N영업일
- 재수정 기준: 미반영 재작업은 미차감, 방향 변경은 신규 1회 또는 CR
- 경미한 수정 범위: 문구, 오탈자, 버튼명, 색상, 라운딩, 간격, 이미지 교체

### 하자

- 하자 정의: 계약 범위 기능 불능 또는 검수기준 미달
- 재현 절차 필요 여부: 필요
- 하자 접수 후 대응 개시: 1영업일 / 2영업일 / 3영업일
- 하자보수 기간: 검수완료 후 0개월 / 1개월 / 2개월 / 3개월 / 직접 입력
- 하자보수와 무상수정 차감: 미차감

### 변경요청

- CR 대상: 신규 기능, 신규 화면, 데이터 구조 변경, 외부 연동, 운영정책 변경, 승인사항 번복
- CR 착수 조건: 견적과 일정 서면 승인 후 착수
- 최소 과금 단위: 0.5시간 / 1시간 / 반일 / 1일
- 시간 단가 또는 일 단가
- 일정 연장 방식: 추가 작업기간만큼 자동 연장 / 별도 합의

### 검수/대금

- 검수 기간: 산출물 제출 후 N영업일
- 반려 방식: 구체 사유 서면 제출
- 미응답 시 승인간주: 사용 / 미사용
- 대금 지급 방식: 선금/중도금/잔금 / 마일스톤 / 검수 후 일괄
- 지급 기한: 세금계산서 또는 청구서 수령 후 N일

### 견적

- 견적 row: ID, 분류, 항목명, 연결 요구사항, 단위, 수량, 단가, 금액, 전제조건, 완수 조건, 테스트 조건, 부수 항목
- 변경요청 단가: 항목, 최소 단위, 단가, 기준
- 견적 전제조건: 제공 자료, 승인 상태, 외부 API 준비, 데이터 정리 상태
- 완수 조건: 해당 row가 완료되었다고 볼 조건
- 테스트 조건: 검수 때 확인할 테스트 방식과 시나리오
- 부수 항목: 큰 row 안에 포함되는 세부 작업
- 줄이는 분쟁: 각 row가 줄이는 분쟁 유형

### 기능별 구현·디자인 명세

- 기능 ID와 기능명
- 연결 요구사항 ID와 연결 견적 row
- 진입점: 화면, 버튼, URL, 메뉴, 알림, 외부 링크
- 사용자/권한: 일반 사용자, 관리자, 비회원, 운영자 등
- 디자인 요건 충족 여부: 충족 / 미충족 / 해당 없음 / 확인 필요
- 상태별 화면: 기본, 로딩, 빈 상태, 오류, 권한 없음, 완료
- 디자인 플로우: Mermaid 원문을 기준으로 작성
- SVG 플로우: Mermaid 렌더링 결과가 있을 때만 첨부
- 최종 결과: 화면 이동, 저장 결과, 알림, 데이터 변경, 관리자 반영 결과
- 완료 기준과 테스트 조건
- 줄이는 분쟁: 진입 경로, 화면 상태, 완료 결과, 디자인 선호 변경 분쟁

### 착수자료/검수실행/리스크 검토

- 착수자료 확정표: PRD, 기능정의서, 화면설계서, 디자인 기준, 검수기준의 제공 상태와 기능별 착수 가능 여부
- 검수실행 기록표: 테스트 ID, 수행 환경, 기대 결과, 실제 결과, 증거 링크, 반려 사유
- 계약서 리스크 검토표: 누락 조항, 모호한 조항, 위험도, 추천 보완 방향, 처리 상태
- 이 세 문서는 긴 행 단위 점검이므로 XLSX를 기본 형식으로 둔다.

### 권리/인도


- 소스코드 인도: 없음 / 최종 납품 시 / 마일스톤별 / 저장소 초대
- 관리자 계정/API 키 인도
- DB 스키마 인도
- 수급자 사전보유자산 제외
- 오픈소스/외부 라이브러리 귀속 분리

## Presets

### 발주자

- 검수기준 필수
- 납품물과 소스코드 인도 범위 명확화
- 하자 대응 기한 2영업일
- 검수기간 5영업일
- 반려 사유 서면 제출

### 수급자

- 확정된 자료가 있는 기능만 착수
- 무상수정 총량 제한
- 신규 기능/화면/데이터 변경은 CR
- 회의록 승인 전 구두 발언 효력 없음
- 대면회의 횟수와 초과 비용 명시

### 중립

- 전체 무상수정 2회 또는 마일스톤별 1회
- 검수기간 5영업일
- 미응답 시 승인간주
- 하자는 무상수정 미차감
- CR은 비용·일정 서면 승인 후 착수

## Output Rules

- 한쪽에게만 유리한 조항처럼 쓰지 않는다.
- “수요자 이점 / 공급자 이점 / 균형안”을 함께 제시한다.
- 모호한 값은 기본 프리셋을 넣되 `확인 필요`로 표시한다.
- 법률상 단정 표현을 피한다.
- 사용자용 산출물은 선택형 입력값 기반 표 형태 DOCX/XLSX 계약문서 패키지로 구성한다.
- 긴 체크표, 견적 row, 검수 실행, 리스크 검토는 XLSX를 우선한다.
- 기능별 구현·디자인 명세는 `진입점 → 플로우 → 최종 결과 → 검수 연결` 순서로 작성한다.
- Mermaid를 원본으로 두고 SVG는 첨부 산출물로만 취급한다.
- 문서별 작성법은 `references/documents/{문서번호-이름}/recipe.md`를 기준으로 한다.
- 전체 목록이 필요할 때만 `references/documents/index.md`를 읽는다.
- 스킬의 작동은 `npm run audit:publish`와 스킬 validator로 입증한다.
- 참고문헌은 확인 가능한 링크와 함께 별도 문서로 둔다.
