# Contributing

이 프로젝트는 SW 외주개발계약에서 반복되는 분쟁을 줄이기 위한 공개 템플릿과 검사기를 만듭니다.

## 우선순위

1. SW 개발용역
2. 유지보수/운영계약
3. 디자인 포함 개발
4. 콘텐츠/디자인 단독 용역

## PR 원칙

- 법률 자문처럼 쓰지 않는다.
- 조항은 실제 계약에 바로 쓰기보다 검토 초안으로 표시한다.
- 확인 가능한 법령, 지침, 계약 양식, 분쟁사례를 우선한다.
- 특정 회사나 사건의 비공개 정보를 포함하지 않는다.
- 템플릿은 수급자와 발주자 모두가 이해할 수 있는 문장으로 쓴다.
- 생성된 HWP/ZIP은 커밋하지 않는다.
- 공개 샘플은 전부 가상 상황으로 작성한다.
- 외부 실무 글은 패턴만 일반화하고, 조항 출처처럼 직접 싣지 않는다.
- 새 문항이나 조항은 어떤 분쟁을 줄이는지 함께 설명한다.

## PR 전 점검

```bash
npm run check
npm run audit:publish
```

`npm run audit:publish`는 공개 샘플 3종을 모두 빌드하고, 문서 무결성, 선택 별첨 포함 여부, 견적 row 연결, 민감 문자열 포함 여부를 검사합니다.

## 공개 금지

- 실제 계약 입력값
- 실고객, 실프로젝트, 담당자 정보
- 접속 정보와 환경 파일
- 업로드 원본 문서
- 생성된 HWP/ZIP
- 로컬 assistant/runtime 상태

## 좋은 PR 예시

- 검수기준 예시 추가
- 변경요청서 항목 개선
- 대면회의 조항 개선
- 리스크 룰 추가
- HWP 표/문단 품질 개선
- 변호사 리뷰를 반영한 조항 경계 개선
- 참여자 니즈를 입력 항목이나 문서 표로 반영

## 리뷰 기준

리뷰어는 [PR Review Checklist](docs/pr-review-checklist.md)를 기준으로 봅니다. 특히 계약 문구 변경 PR은 양측 효과, 줄이는 분쟁, 견적 row 연결 여부를 같이 확인합니다.

## 니즈 수집

로드맵은 [Roadmap](docs/roadmap.md), 변호사 참여 방식은 [Lawyer Participation](docs/lawyer-participation.md), 참여자 니즈 수집 기준은 [Stakeholder Needs Collection](docs/stakeholder-needs.md)에 있습니다.

니즈 제출은 GitHub의 `Stakeholder need` 이슈 템플릿을 사용합니다. 실제 계약 정보가 아니라 반복되는 불편과 필요한 입력값을 적어야 합니다.
