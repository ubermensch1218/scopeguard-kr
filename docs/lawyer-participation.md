# Lawyer Participation

이 문서는 변호사가 이 프로젝트에 참여할 때 어떤 방식으로 검토하고, 어떤 산출물을 남기면 좋은지 정리합니다.

## 프로젝트 목적

이 프로젝트는 분쟁이 발생한 뒤 승패를 예측하는 도구가 아닙니다.

목적은 SW 외주개발계약에서 반복되는 분쟁을 줄이기 위해, 계약 전 합의해야 할 항목을 구조화하고 그 결과를 계약서와 부속서류에 반영하는 것입니다.

## 변호사에게 필요한 역할

| 역할 | 검토 대상 | 결과물 |
|---|---|---|
| 조항 리뷰 | 계약서 본문, 별첨, 템플릿 문구 | 유지 / 수정 / 삭제 / 전문가 검토 필요 |
| 쟁점 분류 | 하자, 무상수정, 변경요청, 검수, 대금, 권리귀속 | 분류 기준과 예외 조건 |
| 균형성 검토 | 발주자와 수급자에게 미치는 효과 | 발주자 이점, 수급자 이점, 균형안 |
| 입력 항목 검토 | 사용자가 선택하는 문항과 옵션 | 필수 입력 / 선택 입력 / 위험 입력 |
| 참고문헌 검토 | 공개 링크와 문서 설명 | 참고 가능 여부와 주의 문구 |

## 우선 검토할 쟁점

| 우선순위 | 쟁점 | 확인할 질문 |
|---|---|---|
| 1 | 하자와 변경요청의 경계 | 기능 불능, 검수 기준 미달, 선호 변경, 신규 기능을 어떻게 나눌 것인가 |
| 2 | 무상수정 횟수 | 수정 1회를 통합 요청 묶음으로 볼 수 있는가 |
| 3 | 검수와 승인 처리 | 검수 기한, 반려 방식, 미응답 처리 문구가 과도하지 않은가 |
| 4 | 대면회의와 구두요청 | 회의록 승인 전 구두 요청의 효력을 어떻게 제한할 것인가 |
| 5 | 견적 row | 전제조건, 완수 조건, 테스트 조건이 계약 범위를 충분히 잠그는가 |
| 6 | 권리귀속 | 프로젝트 전용 산출물, 사전보유자산, 오픈소스, 외부 API를 어떻게 분리할 것인가 |
| 7 | 개인정보/보안 | 처리 항목, 책임 범위, 삭제 요청, 접근 권한을 어떻게 문서화할 것인가 |

## 리뷰 결론 형식

PR 또는 이슈에서는 아래 형식으로 의견을 남깁니다.

| 항목 | 값 |
|---|---|
| 대상 문서 | 계약서 / RFP / 검수기준표 / 수정요청서 / 변경요청서 / 별첨 |
| 대상 문구 | 검토할 문장 또는 표 항목 |
| 결론 | 유지 / 수정 / 삭제 / 전문가 검토 필요 |
| 이유 | 과도한 표현, 누락된 조건, 양측 불균형, 표현 모호성 등 |
| 제안 문구 | 대체 문구 또는 입력 옵션 |
| 줄이는 분쟁 | 이 문구가 줄이는 분쟁 유형 |

## 좋은 리뷰 예시

| 좋은 리뷰 | 이유 |
|---|---|
| “검수 반려는 요구사항 ID, 재현 절차, 기대 결과, 실제 결과를 포함해야 한다” | 반려 사유를 구체화해 막연한 반려를 줄인다. |
| “디자인 선호 변경은 선택 별첨에 포함된 경우에만 수정횟수 산정 기준으로 삼는다” | 선택 조항과 기본 조항을 분리한다. |
| “추가 대면회의는 포함 횟수, 기준 시간, 초과 과금 단위를 나누어 입력해야 한다” | 회의 자체가 과업처럼 확장되는 분쟁을 줄인다. |

## 피해야 할 리뷰

- 실제 사건이나 고객 정보를 포함한 사례
- 특정 당사자에게만 유리한 일방 문구
- 법률 판단을 단정하는 표현
- 생성 계약서에 참고 링크를 조항 근거처럼 넣는 방식
- 문구만 고치고 어떤 분쟁을 줄이는지 설명하지 않는 방식

## 참여 방법

1. `Clause or rule review` 이슈를 연다.
2. 어떤 분쟁을 줄이려는지 먼저 쓴다.
3. 대상 문서와 제안 문구를 적는다.
4. 발주자와 수급자에게 미치는 효과를 함께 적는다.
5. 필요하면 PR로 문구 또는 데이터 변경을 제안한다.
6. PR에서는 `npm run audit:publish` 결과를 남긴다.
