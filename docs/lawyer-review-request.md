# 변호사 검토 요청서

scopeguard-kr은 법률 자문 도구가 아니라 SW 외주개발계약의 운영 리스크를 줄이기 위한 오픈 프로젝트입니다.

이 문서는 발주자, 개발사, 프리랜서, 스타트업 실무자가 변호사에게 계약서 검토를 요청하기 전에 어떤 계약서를 먼저 맡길지, 어떤 자료를 함께 보낼지 정리하는 초안입니다.

## 사용 원칙

아래 중 하나라도 해당하면 변호사 검토 우선순위를 높입니다.

| 기준 | 검토가 필요한 이유 |
|---|---|
| 손실을 감당하기 어려운 금액 | 대금, 환불, 위약벌, 지체상금, 손해배상 한도 조항이 실제 현금흐름을 흔들 수 있음 |
| 소스코드, 디자인, 데이터, 모델, 콘텐츠 권리 | 한 번 잘못 넘기면 재사용, 유지보수, 판매, 투자 실사에서 문제가 커짐 |
| 개인정보 또는 고객 데이터 처리 | 위탁, 제3자 제공, 보관, 삭제, 재위탁, 침해 통지 책임을 문서로 나눠야 함 |
| 상대방 양식으로 체결 | 우리 업무 방식과 맞지 않는 검수, 해지, 책임 조항이 숨어 있을 수 있음 |
| 공공, 대기업, 공적 지원사업 | 감사, 환수, 보안, 산출물 귀속, 하도급 제한이 일반 계약보다 무거움 |
| 장기 계약 또는 자동갱신 | 해지, 가격 변경, SLA, 데이터 반출 조건을 놓치면 빠져나오기 어려움 |
| 사람을 묶는 계약 | 고용, 프리랜서, 경업금지, 영업비밀, 스톡옵션은 분쟁 시 감정 비용도 큼 |

## 체크리스트로 먼저 걸러도 되는 계약

아래에 모두 해당하면 변호사 검토보다 내부 체크리스트가 먼저일 수 있습니다.

| 조건 | 확인할 것 |
|---|---|
| 소액이고 1회성 | 환불, 취소, 납품일, 세금계산서만 확인 |
| 개인정보나 고객 데이터가 없음 | 이메일, 연락처, 접속 로그, 파일 업로드가 정말 없는지 확인 |
| IP 이전이 없음 | 산출물 사용권이나 재사용 제한이 숨어 있지 않은지 확인 |
| 자동갱신과 위약벌이 없음 | 해지 통지 기간, 최소 사용기간, 위약금 확인 |
| 상대방 기본약관을 그대로 수락해도 사업 영향이 작음 | 서비스 중단, 가격 변경, 데이터 반출 가능성 확인 |

## 변호사에게 보내는 요청 문구

아래 문구를 복사해서 계약서와 함께 보냅니다.

```text
안녕하세요. 아래 계약서의 체결 전 리스크 검토를 부탁드립니다.

저희 입장은 [발주자 / 수급자 / 서비스 제공자 / 서비스 이용자 / 투자자 / 피투자사]입니다.
이번 검토의 목적은 소송 가능성 예측이 아니라, 체결 전에 조항의 모호함과 과도한 책임을 줄이는 것입니다.

우선 검토해 주실 부분은 다음입니다.

1. 대금, 검수, 해지, 지체상금, 손해배상 한도
2. 하자, 무상수정, 변경요청의 경계
3. 소스코드, 산출물, 사전보유자산, 오픈소스의 권리귀속
4. 개인정보, 데이터, 보안, 재위탁, 삭제/반환 책임
5. 상대방에게 일방적으로 유리하거나 저희 업무 방식과 맞지 않는 조항
6. 계약서에 추가해야 할 별첨, 증빙, 승인 절차

가능하면 검토 결과를 아래 형식으로 정리해 주세요.

- 조항 위치:
- 위험도: 높음 / 중간 / 낮음
- 문제:
- 제안 문구:
- 상대방과 협상할 때 반드시 지킬 선:
- 양보 가능 조건:
```

## SW 외주개발계약 검토 요청 항목

| 쟁점 | 변호사에게 확인할 질문 | Scopeguard에서 붙일 자료 |
|---|---|---|
| 하자와 경미한 수정의 경계 | 기능 불능, 검수기준 미달, 선호 변경, 신규 기능을 어떻게 나눌 것인가 | 하자신고서, 수정요청서, 변경요청서 |
| 무상수정 총량 | 수정 1회를 통합 요청 묶음으로 볼 수 있는가 | 수정횟수 산정표 |
| 검수와 승인 처리 | 검수 기한, 반려 방식, 미응답 승인간주가 과도하지 않은가 | 검수기준표, 검수실행 기록표, 납품확인서 |
| 구두 요청과 회의록 | 회의록 승인 전 구두 발언의 효력을 제한할 수 있는가 | 회의록 승인서 |
| 견적 row | 전제조건, 완수 조건, 테스트 조건이 계약 범위를 충분히 고정하는가 | 견적산정표, RFP/과업내용서 |
| 변경요청 | 신규 기능, 신규 화면, 데이터 구조 변경, 외부 연동을 유상 CR로 처리할 수 있는가 | 변경요청서 |
| 권리귀속 | 프로젝트 전용 산출물, 사전보유자산, 오픈소스, 외부 API를 어떻게 분리할 것인가 | 권리귀속/소스코드 인도목록 |
| 운영비 | 클라우드, API, 모델, 계정, 도메인 비용을 누가 부담하는가 | 운영비/API/계정 인수인계서 |
| 개인정보/보안 | 처리 항목, 재위탁, 삭제, 접근권한, 사고 통지를 어떻게 문서화할 것인가 | 개인정보/보안 요구사항 |

## 첨부자료 묶음

| 묶음 | 포함할 파일 | 보내기 전 확인 |
|---|---|---|
| 계약 원문 | 상대방 계약서, 우리 수정안, 별첨, 약관 링크 | 최신본인지 확인하고 파일명에 날짜를 붙임 |
| 과업 범위 | 제안서, 견적서, RFP, 기능정의서, 화면설계서, Figma 링크 | 요구사항 ID와 견적 row가 연결되어 있는지 확인 |
| 검수/납품 | 검수기준표, 테스트 시나리오, 납품확인서, 승인 절차 | 반려 사유와 증빙 방식이 적혀 있는지 확인 |
| 변경/수정 | 수정요청서, 변경요청서, 회의록 승인서 | 무상수정과 유상 변경요청이 분리되어 있는지 확인 |
| 권리/보안 | 권리귀속표, 소스코드 인도목록, 개인정보/보안 요구사항 | 고객정보, API 키, 계정 비밀번호는 제거 |
| 사업 조건 | 결제 일정, 세금계산서 조건, 해지 조건, SLA | 대금 지급과 산출물 승인이 연결되어 있는지 확인 |

## PR 방식

변호사가 이 저장소에 의견을 남길 때는 아래 방식으로 반영합니다.

- 계약 조항 문구 개선은 `templates/`에 제안
- 법령/지침/사례 근거는 `docs/legal-references.md`에 출처 포함
- 룰 추가는 `assets/app.js`의 `riskRules` 또는 추후 `data/risk-rules.json`에 제안
- 긴 검토표, 견적 row, 검수 실행, 리스크 검토는 XLSX 형태를 우선

## 주의

검토 의견은 일반 정보로 공개되며, 개별 사건에 대한 법률 자문으로 표시하지 않습니다. 실제 계약 체결 여부, 조항의 효력, 분쟁 가능성은 변호사에게 별도 자문으로 확인해야 합니다.
