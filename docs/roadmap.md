# Roadmap

이 로드맵은 SW 외주개발계약에서 분쟁을 줄이는 계약문서 작성기를 단계적으로 키우기 위한 공개 작업 순서입니다.

## 원칙

- 소송 자동화가 아니라 계약 전후의 합의 기준을 선명하게 만든다.
- 체크리스트에서 멈추지 않고 계약서, RFP, 검수기준표, 수정요청서, 변경요청서, 견적산정표에 반영한다.
- 견적 row를 중심에 둔다. 금액, 전제조건, 완수 조건, 테스트 조건, 부수 항목을 끊지 않는다.
- 공개 샘플은 전부 가상 상황으로 둔다.
- 계약 문구는 전문가 검토가 가능한 초안으로 표시한다.

## v0.1 현재

| 영역 | 상태 |
|---|---|
| DOCX 생성기 | 기본 계약문서 패키지 생성 |
| 샘플 입력 | 기본형, 디자인형, 운영/API형 |
| 선택 별첨 | 디자인/UX 수정 범위, 수정횟수 산정표 |
| 공개 audit | 샘플 빌드, DOCX 무결성, 문자열 점검 |
| GitHub 방어막 | CI, PR 템플릿, 이슈 템플릿, CODEOWNERS, 보안 안내 |

## v0.2 입력 구조 고정

| 목표 | 필요한 작업 |
|---|---|
| 계약 입력 구조 안정화 | `contract-input` JSON Schema 작성 |
| 견적 row 검증 | 수량, 단가, 금액 산식 확인 |
| 요구사항 연결 확인 | `linkedRequirementIds`가 실제 요구사항에 있는지 확인 |
| 선택 별첨 검증 | 입력값과 생성 문서 수 일치 확인 |
| 샘플 확장 | 유지보수형, 디자인 단독형 샘플 추가 |

## v0.3 전문가 검토 패키지

| 목표 | 필요한 작업 |
|---|---|
| 변호사 리뷰 흐름 정리 | 문구별 검토 포인트와 리뷰 결론 양식 추가 |
| 조항별 쟁점 표시 | 하자, 무상수정, 변경요청, 검수, 회의록, 권리귀속 분리 |
| 균형안 표기 | 발주자 이점, 수급자 이점, 균형안 표시 |
| 검토 요청 이슈화 | clause review 이슈 템플릿 고도화 |
| 전문가 피드백 반영 | 계약 문구 변경 PR 기준 정리 |

## v0.4 입력 UI 작성 흐름

| 목표 | 필요한 작업 |
|---|---|
| 첫 선택 단순화 | 발주자 / 수급자 / 중립 관점 선택 |
| 계약 유형 선택 | 기본형 / 디자인 포함 / 운영·API 포함 / 유지보수형 |
| 견적 row 입력 | 전제조건, 완수 조건, 테스트 조건, 부수 항목 입력 |
| 오버레이 표시 | 어떤 문항이 어떤 분쟁을 줄이는지 표시 |
| DOCX 생성 연결 | 입력값 저장 후 계약문서 패키지 생성 |

## v0.5 문서 품질 개선

| 목표 | 필요한 작업 |
|---|---|
| DOCX 스타일 개선 | 제목, 표, 여백, 페이지 구성 개선 |
| 문서별 책임 분리 | 계약서 본문과 별첨의 중복 줄이기 |
| 조항 선택 옵션 | 디자인/UX, 수정횟수, 운영비, 회의비 조항 선택화 |
| 문서 검토 화면 | 생성 전 누락값과 위험 항목 표시 |
| 내보내기 개선 | DOCX 패키지 외 별도 문서 선택 생성 |

## v0.6 문서 파이프라인 확장

| 목표 | 필요한 작업 |
|---|---|
| HWPX 연계 검토 | 기존 계약 양식 변환과 비교 흐름 검토 |
| 문서 비교 | 이전 계약서와 새 입력값 차이 확인 |
| 참고문헌 분리 | 생성 계약문서와 참고 링크 문서 분리 유지 |

## 참여가 필요한 영역

| 참여자 | 필요한 도움 |
|---|---|
| 변호사 | 조항 경계, 문구 위험, 양측 균형 검토 |
| 개발사/프리랜서 | 실제로 자주 터지는 요청 유형과 견적 row 구조 제안 |
| 발주자 | 이해하기 쉬운 입력 문항과 검수 기준 피드백 |
| 디자이너/PM | 화면설계, 디자인 변경, UX 개선 요청 분류 피드백 |
| 문서 자동화 개발자 | DOCX/HWPX 생성, JSON Schema, UI 작성 흐름 개선 |
