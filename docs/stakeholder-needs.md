# Stakeholder Needs Collection

이 문서는 각 참여자가 무엇을 필요로 하는지 수집하기 위한 기준입니다. 목표는 계약서를 더 길게 만드는 것이 아니라, 분쟁이 커지기 전에 합의해야 할 항목을 정확히 묻는 것입니다.

## 수집 원칙

- 실제 계약 정보는 공개 이슈에 쓰지 않는다.
- 가상 사례나 익명화된 반복 패턴만 공유한다.
- “원하는 조항”보다 “줄이고 싶은 분쟁”을 먼저 적는다.
- 계약서 본문, RFP, 검수기준표, 수정요청서, 변경요청서, 견적산정표 중 어디에 반영될지 함께 적는다.
- 발주자와 수급자에게 각각 어떤 효과가 있는지 적는다.

## 참여자별 니즈

| 참여자 | 자주 필요한 것 | 제품에 반영할 입력 |
|---|---|---|
| 발주자 | 납품물이 무엇인지, 언제 검수하는지, 하자와 미완료를 어떻게 나누는지 | 요구사항 ID, 검수 기준, 반려 방식, 납품확인서 |
| 수급자 | 무상수정 총량, 변경요청 전환 기준, 자료 제공 지연 시 일정 처리 | 수정횟수, CR 기준, 착수자료 상태, 일정 연장 기준 |
| 변호사 | 문구 위험, 양측 균형, 조항의 예외 조건 | 조항 리뷰 결론, 위험 표시, 대체 문구 |
| PM/기획자 | 기능 범위, 화면 흐름, 승인된 자료 변경 관리 | PRD, 기능정의서, 화면설계서, 사용자 흐름 |
| 디자이너 | 선호 변경과 하자의 구분, 디자인 상태와 반응형 기준 | 디자인 기준, 화면 상태, 라운딩/간격/컴포넌트 기준 |
| 개발사/프리랜서 | 견적 row의 전제조건과 추가 작업 단가 | row별 전제조건, 완수 조건, 테스트 조건, 부수 항목 |
| 플랫폼 운영자 | 공개 이슈와 샘플에서 실계약 정보가 섞이지 않는 구조 | 이슈 템플릿, audit, 공개 경계 문서 |

## 니즈 제출 양식

| 항목 | 작성 내용 |
|---|---|
| 참여자 유형 | 발주자 / 수급자 / 변호사 / PM / 디자이너 / 개발자 / 플랫폼 운영자 |
| 줄이고 싶은 분쟁 | 예: “디자인 더 예쁘게” 요청이 무상수정인지 계속 다툼 |
| 현재 불편 | 계약서나 협업 과정에서 어디가 모호한지 |
| 필요한 입력값 | 예: 화면 ID, 디자인 기준, 수정 요청 회차, 승인 자료 링크 |
| 반영 문서 | 계약서 / RFP / 검수기준표 / 수정요청서 / 변경요청서 / 견적산정표 |
| 발주자 효과 | 발주자가 얻는 명확성 |
| 수급자 효과 | 수급자가 얻는 명확성 |
| 검토 필요성 | 변호사 검토 필요 / 실무자 검토 필요 / 개발 구현 필요 |

## 수집할 질문 예시

| 영역 | 질문 |
|---|---|
| 착수자료 | PRD, 기능정의서, 화면설계서, 검수기준이 언제 확정되는가 |
| 견적 | 이 row의 금액은 어떤 전제조건에서 유효한가 |
| 완수 조건 | 어떤 상태가 되면 이 row가 완료된 것으로 볼 수 있는가 |
| 테스트 조건 | 검수 때 어떤 시나리오를 확인할 것인가 |
| 수정 | 수정 1회를 어떤 제출 단위로 볼 것인가 |
| 디자인 | 라운딩, 색상, 간격, 컴포넌트 기준이 확정되어 있는가 |
| 회의 | 대면회의 포함 횟수와 초과 비용이 있는가 |
| 변경요청 | 신규 기능, 신규 화면, 승인사항 번복을 어떻게 전환할 것인가 |
| 권리귀속 | 프로젝트 전용 산출물과 사전보유자산이 분리되어 있는가 |
| 운영 | 클라우드, 외부 API, 도메인, 알림 비용은 누가 부담하는가 |

## 반영 우선순위

| 우선순위 | 기준 |
|---|---|
| 높음 | 반복 분쟁이 많고, 입력값으로 계약서/별첨에 바로 반영 가능한 항목 |
| 중간 | 사례는 많지만 계약 문구 검토가 필요한 항목 |
| 낮음 | 특정 업종이나 특정 프로젝트에만 적용되는 항목 |
