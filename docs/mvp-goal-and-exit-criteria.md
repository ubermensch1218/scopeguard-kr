# MVP Goal and Exit Criteria

## 현재 goal

수급자/발주자가 견적 row를 입력하면, 각 row의 전제조건, 완수 조건, 테스트 조건, 부수 항목이 자동으로 계약서 조항과 부속서류에 반영되는 SW 외주계약 작성기 MVP를 만든다.

한 줄 정의:

> 견적서 기반 SW 외주계약 자동 작성기

저장소 설명:

> scopeguard-kr: 견적 row를 기준으로 SW 외주계약서와 부속서류를 생성하는 분쟁 예방 계약 작성기

## 핵심 사용자 경험

사용자는 법률 문장을 직접 작성하지 않는다. 사용자는 버튼, 셀렉트, 숫자 입력, 표 입력으로 계약 파라미터와 견적 row를 채운다.

각 견적 row는 다음 값을 가진다.

| 항목 | 용도 |
|---|---|
| ID | 계약서, RFP, 검수표, 변경요청서에서 같은 row를 추적 |
| 항목명 | 견적 대상 작업 |
| 연결 기능 | 요구사항 ID 또는 작업 묶음 |
| 단위/수량/단가/금액 | 계약금액과 변경요청 기준 |
| 전제조건 | 해당 금액과 일정이 유지되는 조건 |
| 완수 조건 | 납품 완료 판단 기준 |
| 테스트 조건 | 검수 방식 |
| 부수 항목 | row에 포함되는 세부 작업 |
| 줄이는 분쟁 | 이 문항이 줄이는 분쟁 |

## 각 입장에서 느낄 불편함

| 입장 | 예상 불편함 | 제품 대응 |
|---|---|---|
| 수급자 | 입력할 항목이 많고 견적 전제조건을 하나씩 쓰기 번거로움 | 기본 row 예시, row 복제, 프리셋, 필수 누락 체크 |
| 수급자 | 견적서에 수량, 단가, 금액이 발주자에게 보이므로 row별 전제조건이 빠지면 단가가 무한 범위로 해석될 수 있음 | 모든 견적 row에 전제조건, 완수 조건, 테스트 조건, 부수 항목을 묶어 계약 범위를 잠근다 |
| 수급자 | 발주자가 자료를 안 준 상태에서 작성이 멈춤 | 착수자료 상태를 `발주자 제공 필요`로 표시하고 일정 연장 조항에 반영 |
| 발주자 | 개발 용어와 row 단가 구조가 어렵게 느껴짐 | 문항 오버레이에 “왜 묻는지/줄이는 분쟁/반영 문서” 표시 |
| 발주자 | 수급자에게 유리한 조항처럼 보일 수 있음 | 합의표에 수요자 이점, 공급자 이점, 균형안을 같이 표시 |
| 발주자 | 완수 조건과 테스트 조건을 어떻게 써야 할지 모름 | 기능별 예시 row와 테스트 조건 예시 제공 |
| 검토자 | 문서가 많아 전체 구조를 한 번에 보기 어려움 | 입력값 요약표와 견적산정표를 첫 검토 문서로 사용 |
| 검토자 | 실제 계약서에 어떤 row가 어디 반영됐는지 추적이 필요함 | 모든 핵심 문서에 견적 row ID를 유지 |
| 운영자 | 웹 UI 표가 모바일에서 길고 무거움 | 모바일은 가로 스크롤을 허용하고, 이후 row별 편집 패널로 개선 |

## 견적서 공개 범위

견적서의 수량, 단가, 금액은 발주자에게 제출되는 값이다. 공개 여부를 숨기는 문제가 아니라, 그 금액이 어떤 전제조건, 완수 조건, 테스트 조건, 부수 항목에 묶이는지가 핵심이다.

웹 서비스 단계에서 입력 로그나 row 패턴을 제품 개선에 활용할 수는 있다. 다만 현재 정적 MVP는 서버 수집을 하지 않으며, 서비스형으로 전환할 때는 입력 데이터의 수집 범위, 익명화 기준, 민감정보 제외 기준을 별도 제품 요구사항으로 둔다.

## 종료 조건

MVP는 아래 조건을 모두 만족해야 종료한다.

| 조건 | 확인 방법 |
|---|---|
| 발주자/수급자/중립 관점 선택 가능 | `index.html`의 `data-role` 버튼 3개 확인 |
| 견적 row 기반 노코드 입력 UI 제공 | `#estimate-rows` 표, `#add-estimate-row` 버튼 확인 |
| 견적 row 필수값 입력 가능 | ID, 항목, 연결 기능, 단위, 수량, 단가, 전제조건, 완수 조건, 테스트 조건, 부수 항목, 줄이는 분쟁 입력 필드 확인 |
| 문항별 “줄이는 분쟁” 오버레이 제공 | `#overlay-list`와 `overlayItems` 확인 |
| 누락 체크 제공 | `#writer-checks`와 `renderWriterChecks()` 확인 |
| 계약서 초안 HWP 생성 | `03_SW_개발용역계약서_초안.hwp` 생성 및 HWP 정보 검사 |
| RFP/과업내용서 HWP 생성 | `02_RFP_과업내용서.hwp` 생성 및 HWP 정보 검사 |
| 검수기준표 HWP 생성 | `04_검수기준표.hwp` 생성 및 HWP 정보 검사 |
| 견적산정표 HWP 생성 | `15_견적산정표.hwp` 생성 및 HWP 정보 검사 |
| 수정요청서/변경요청서 등 부속서류 ZIP 생성 | `99_전체_산출물_보관용.zip`에 기본 18개 HWP와 14개 XLSX 포함 |
| 견적 row가 핵심 문서에 반영 | RFP, 계약서, 검수기준표, 대금표, 견적산정표의 XML에서 `EST-` 확인 |
| 브라우저 콘솔 에러 없음 | Playwright 콘솔 에러 확인 |

## 다음 개선 목표

MVP 이후의 다음 goal은 다음 중 하나다.

1. 견적 row 편집을 표가 아니라 row별 패널로 개선한다.
2. 입력값 JSON 내보내기/불러오기를 붙인다.
3. 견적 row 로그/분석 이벤트 스키마를 설계한다.
4. 기존 계약서 업로드 후 누락된 row와 조항을 역으로 표시한다.
5. HWP 문서 품질 검증을 강화한다.
6. 변호사/PM 검토 코멘트 흐름을 붙인다.
