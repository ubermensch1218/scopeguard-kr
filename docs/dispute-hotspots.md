# 분쟁 핫스팟

이 문서는 Scopeguard가 우선 잠가야 할 분쟁 지점을 정리한다. 법률 판단이 아니라 계약 운영 중 실제로 다툼이 커지기 쉬운 구석을 제품 문항과 산출물로 연결하기 위한 기준이다.

## 먼저 봐야 할 지점

| 우선 | 지점 | 왜 커지는가 | 잠그는 문서 |
|---|---|---|---|
| 1 | 착수자료 미확정 | PRD, 기능정의서, 화면설계, 데이터, 검수 기준이 없는데 개발이 시작되면 일정 지연과 범위 변경이 한꺼번에 생긴다. | 착수자료 확정표, RFP/과업내용서, 계약서 착수 조건 |
| 2 | 검수 기준 부재 | 납품 후 “완료”와 “마음에 들지 않음”이 섞인다. 반려 사유가 구체적이지 않으면 잔금과 하자보수가 같이 막힌다. | 검수기준표, 검수실행 기록표, 납품확인서 |
| 3 | 수정과 변경요청 혼재 | 문구 수정, 디자인 선호 변경, 신규 기능이 한 요청서에 들어가면 무상인지 유상인지가 흐려진다. | 수정요청서, 변경요청서, 수정횟수 산정표 |
| 4 | 회의록 없는 구두요청 | 회의 중 말한 내용이 계약 범위처럼 주장된다. 누가 언제 승인했는지 증거가 약해진다. | 회의록 승인서, 변경요청서 |
| 5 | 권리귀속과 NDA 경계 | 소스코드, 사전보유자산, 피드백, 공동개발 결과물, 비밀정보의 사용 범위가 뒤늦게 문제 된다. | 권리귀속/소스코드 인도목록, 변호사 검토 요청서, 계약서 리스크 검토표 |
| 6 | 운영비와 계정 인수 | API, 모델, 도메인, 문자, 클라우드, 관리자 계정이 납품 뒤에도 비용과 책임으로 남는다. | 운영비/API/계정 인수인계서, 개인정보/보안 요구사항 |
| 7 | 대금과 마일스톤 연결 끊김 | 산출물, 검수, 세금계산서, 잔금 지급 조건이 서로 다른 기준으로 움직인다. | 대금 마일스톤 지급표, 납품확인서 |

## 내가 더 위험하게 보는 케이스

| 케이스 | 위험 신호 | 제품에서 더 강하게 물어야 할 것 |
|---|---|---|
| 외부 API나 모델이 핵심 기능인 프로젝트 | 기능은 납품됐지만 외부 비용, 속도, 장애, 정책 변경이 남는다. | 운영비 부담자, 계정 소유자, 장애 대응 범위, 대체 수단 |
| 디자인 포함 프로젝트 | “예쁘게”, “고급스럽게”, “요즘 느낌” 같은 표현이 검수 기준처럼 쓰인다. | 디자인 기준, 상태별 화면, 레퍼런스, 수정횟수 별첨 포함 여부 |
| 고객 데이터가 들어가는 관리자/운영 도구 | 개발 중 접근권한, 로그, 삭제, 반출 기준이 뒤늦게 나온다. | 처리 항목, 접근자, 보관기간, 삭제/반환, 테스트 데이터 사용 |
| 인수인계가 중요한 프로젝트 | 산출물은 받았지만 계정, DB 스키마, 배포 문서, 권한이 빠진다. | 소스코드 인도, 관리자 계정, 운영 가이드, DB 스키마 |
| 계약 전 미공개 자료를 많이 보여주는 프로젝트 | NDA는 썼지만 무엇이 비밀인지, 무엇에 써도 되는지 애매하다. | 비밀정보 목록, 사용목적, 열람자, 반환/삭제, 잔존지식, 피드백 권리 |
| “유지보수 포함”으로 퉁치는 프로젝트 | 장애 대응, 기능 개선, 운영 질의, 데이터 보정이 모두 유지보수처럼 들어온다. | 유지보수 범위, 응답 시간, 제외 항목, 변경요청 단가 |

## README에 남길 정도의 한 줄 기준

Scopeguard가 닫아야 하는 것은 조항 문구만이 아니다. 실제 분쟁은 “요청이 들어왔을 때 어느 문서로 접수하고, 어떤 기준으로 비용·일정·검수를 다시 판단할지”가 비어 있을 때 커진다.
