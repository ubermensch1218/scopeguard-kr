# PRD: scopeguard-kr

## 1. 제품 정의

`scopeguard-kr`는 SW 외주개발에서 반복되는 분쟁 예방 기준을 실제 계약 조항과 부속서류에 반영해, 계약문서 초안 세트를 DOCX로 제공하는 도구다.

제품의 핵심은 말뿐인 안내문을 주는 것이 아니라, 발주자와 수급자가 같은 기준으로 서명하고 운영할 수 있는 계약서를 만드는 것이다. 내부 점검 룰셋은 사용자에게 따로 주는 문서가 아니라, 계약서 조항과 부속서류에 반영되는 작성 기준이다.

## 2. 문제

SW 외주개발 분쟁은 보통 개발 실패 하나로 시작되지 않는다. 계약 전후의 기준이 열려 있어서 커진다.

- 무엇을 만들지 불명확한 상태에서 착수한다.
- 화면, 기능, 데이터, 검수 기준이 서로 다른 문서에 흩어진다.
- 하자, 경미한 수정, UX 변경, 신규 기능이 한 요청 안에 섞인다.
- 무상수정 횟수는 있지만 “1회”의 기준이 없다.
- 회의 중 발언과 서면 합의의 경계가 없다.
- 대금, 마일스톤, 납품물, 소스코드 인도, 운영비 부담이 뒤늦게 쟁점화된다.

## 3. 목표

MVP의 목표는 다음 하나다.

버튼, 셀렉트, 숫자 입력, 표 입력으로 계약 파라미터를 받고, 분쟁을 줄이는 운영 기준이 조항으로 반영된 DOCX 계약문서 초안 패키지를 생성한다.

부수 목표:

- 발주자가 원하는 결과물과 검수권을 문서화한다.
- 수급자가 부담하는 과업 범위와 변경요청 경계를 문서화한다.
- 양측이 하자, 무상수정, 변경요청을 같은 기준으로 볼 수 있게 한다.
- 기능별 완료 기준과 검수 기준을 문서마다 반복 가능하게 만든다.
- 참고문헌은 별도 문서로 분리하여 실제 확인 가능한 링크를 제공한다.

### 3.1 핵심 방식

내부 룰셋은 다음처럼 계약서와 부속서류에 반영된다.

| 내부 기준 | 계약문서 반영 위치 |
|---|---|
| 착수자료가 확정된 기능만 개발 착수 | 개발 착수 조건 조항, 일정 연장 조항 |
| 포함 범위와 제외 범위 분리 | 계약서 과업 범위 조항, RFP/과업내용서 |
| 기능별 완료 기준 작성 | 검수 조항, 검수기준표, 마일스톤 지급표 |
| 무상수정 횟수와 1회 기준 명확화 | 하자와 무상수정 조항, 수정요청서 |
| 신규 기능·화면·데이터 변경 분리 | 변경요청 조항, 변경요청서 |
| 대면회의·구두요청 효력 제한 | 회의 및 커뮤니케이션 조항, 회의록 승인서 |
| 대금 지급 조건과 검수 연결 | 대금 조항, 마일스톤 지급표 |
| 견적 row와 요구사항 연결 | RFP/과업내용서, 계약서 과업 범위 조항, 대금 지급표, 견적산정표, 변경요청서 |
| 소스코드·외부 자산·사전보유자산 분리 | 권리 귀속 조항, 소스코드 인도목록 |
| 운영비와 외부 서비스 계정 부담 명확화 | 운영비/API/계정 인수인계서 |
| 하자 신고와 재현 절차 명확화 | 하자 조항, 하자신고서 |

## 4. 하지 않는 것

MVP에서 하지 않는다.

- 법률 자문
- 승패 예측
- 자동 분쟁 판단
- 계약서 내용의 최종 적법성 판단
- 대금 결제 또는 에스크로
- 전자서명
- 문서 보관 서비스
- 사건 관리 SaaS
- 자연어 기반 자동 분류

## 5. 사용자

### 5.1 수급자

소규모 개발사, 프리랜서, PM.

원하는 것:

- 무한 수정 방지
- 착수자료 미제공 시 일정 보호
- 신규 기능과 경미한 수정을 분리
- 구두요청을 서면 요청으로 전환
- 대면회의와 추가 작업의 비용 기준 명확화

### 5.2 발주자

초기 스타트업, 연구실, 학교, 기관, 비개발 조직.

원하는 것:

- 원하는 결과물이 계약 문서에 빠지지 않게 하기
- 기능별 검수 기준 확보
- 납품물, 소스코드, 계정, 운영비 부담 명확화
- 하자 발생 시 재현 절차와 대응 기준 확보
- 개발사 변경이나 인수인계 상황에 대비

### 5.3 검토자

변호사, PM, 실무 관리자.

원하는 것:

- 계약 쟁점이 표로 정리된 초안
- 누락 항목을 빠르게 확인할 수 있는 입력값 요약
- 참고문헌 링크 확인
- 조항 문구보다 운영 기준을 먼저 검토

## 6. 핵심 사용자 흐름

### Flow A. 새 계약문서 세트 생성

1. 사용자가 발주자/수급자/중립 관점을 고른다.
2. 사용자가 버튼, 셀렉트, 숫자 입력, 표 입력으로 계약 파라미터를 채운다.
3. 사용자가 견적 row를 기능 ID 또는 작업 묶음과 연결하고, 전제조건·완수 조건·테스트 조건·부수 항목을 입력한다.
4. 제품이 내부 체크리스트로 누락값과 충돌값을 검증한다.
5. 제품이 여러 계약문서 양식에 입력값을 반영한다.
6. 사용자가 DOCX 초안 패키지를 내려받는다.
7. 검토자는 필요한 문구를 수정하거나 삭제한다.

### Flow B. 웹 입력 보조

1. 사용자가 웹에서 발주자/수급자/중립 관점을 고른다.
2. 웹이 각 계약문서에 필요한 입력값을 항목별로 보여준다.
3. 사용자가 버튼과 셀렉트로 대부분의 값을 고르고, 기능 목록과 완료 기준만 표로 입력한다.
4. 웹이 누락 항목과 위험 표현을 화면에서 보여준다.
5. 사용자가 입력값 JSON을 내려받거나 계약문서 패키지를 생성한다.

체크리스트는 사용자가 받는 별도 산출 문서가 아니라, 계약서와 부속서류에 필요한 조항이 빠졌는지 확인하는 내부 기준이다. 화면에는 누락 항목, 충돌 항목, 추가 확인 항목으로 표시하고, 최종 산출물에는 해당 기준이 조항과 표로 반영된다.

## 7. 입력값

MVP 입력값은 `data/contract-input.sample.json`을 기준으로 한다.

필수 그룹:

- `project`: 프로젝트명, 계약 유형, 작성 관점, 요약
- `parties`: 발주자, 수급자, 담당자
- `schedule`: 착수일, 납품일, 지연 처리 기준
- `payment`: 총액, 선금, 중도금, 잔금, 지급 기한
- `estimate`: 견적 row, 연결 요구사항, 단위, 수량, 단가, 금액, 전제조건, 완수 조건, 테스트 조건, 부수 항목, 변경요청 단가
- `startMaterials`: PRD, 기능정의서, 화면설계서, 사용자 흐름, 디자인 기준, 콘텐츠, 데이터/API, 검수 기준
- `scopeIncluded`: 포함 범위
- `exclusions`: 제외 범위
- `requirements`: 요구사항 ID, 분류, 명칭, 설명, 검수 기준, 상태
- `acceptance`: 검수 기간, 반려 방식, 승인 처리
- `revisions`: 무상수정 횟수, 수정 1회 기준, 경미한 수정 범위
- `defects`: 하자 정의, 대응 개시
- `changeRequests`: 변경요청 대상, 착수 조건, 과금 단위, 일정 영향
- `meetings`: 온라인 회의, 대면회의, 추가 회의, 회의록, 구두요청 효력
- `deliverables`: 소스코드, DB 스키마, 관리자 계정, 운영 가이드
- `rights`: 프로젝트 전용 산출물, 사전보유자산, 외부 자산
- `operationCosts`: 클라우드, API, 문자, 도메인, 라이선스 비용 부담
- `privacy`: 개인정보 처리 여부와 처리 항목
- `security`: 접근통제, 로그
- `milestones`: 마일스톤, 기한, 산출물, 지급, 검수 기준
- `optionalAppendices`: 디자인/UX 수정 범위 별첨, 수정횟수 산정표 별첨 포함 여부

입력 UI는 자유 텍스트를 최소화한다.

| 입력 유형 | 쓰는 곳 |
|---|---|
| 버튼 선택 | 작성 관점, 계약 유형, 착수자료 보유 여부, 승인 처리 |
| 셀렉트 | 회의 주기, 수정 방식, 지급 방식, 권리 귀속 방식 |
| 숫자 입력 | 무상수정 횟수, 검수 기간, 회의 포함 횟수, 초과 단가 |
| 표 입력 | 기능 목록, 완료 기준, 견적 row, 마일스톤, 납품물 목록 |
| 짧은 문장 입력 | 프로젝트 요약, 제외 범위, 특약 |

완료 기준은 최소한 다음 칸을 가진다.

| 항목 | 설명 |
|---|---|
| 기능 ID | 요구사항과 검수기준을 연결하는 ID |
| 기능명 | 사용자가 이해할 수 있는 기능 이름 |
| 완료 조건 | 어떤 상태이면 완료로 볼지 |
| 확인 방법 | 화면 확인, 테스트 계정, 데이터 확인, 파일 제출 등 |
| 반려 가능 사유 | 어떤 경우 검수 반려가 가능한지 |
| 제외 사항 | 추가개발 또는 변경요청으로 볼 사항 |

견적 row는 최소한 다음 칸을 가진다.

| 항목 | 설명 |
|---|---|
| 견적 ID | 견적 항목을 식별하는 ID |
| 분류 | 착수/기획, 디자인, 개발, 데이터, 배포, 회의, 변경요청 등 |
| 항목명 | 견적 대상 작업 |
| 연결 요구사항 | 기능 ID 또는 작업 묶음 ID |
| 단위 | 시간, 반일, 일, 건, 화면, 기능 등 |
| 수량 | 산정 수량 |
| 단가 | 단위당 금액 |
| 금액 | 수량과 단가 기준 금액 |
| 전제조건 | 해당 금액이 유지되는 조건 |
| 완수 조건 | 해당 row가 완료되었다고 볼 조건 |
| 테스트 조건 | 발주자와 수급자가 확인할 테스트 방식 |
| 부수 항목 | row에 포함되는 세부 작업 목록 |
| 줄이는 분쟁 | 이 row가 줄이는 분쟁 유형 |

계약 리스크 체크리스트, 화면설계 체크리스트, 입력 누락 체크리스트는 별도 안내문으로 제공하지 않는다. 내부 점검 룰셋과 웹 화면 보조 항목으로 두고, 통과한 내용은 계약서 조항과 부속서류 표에 반영한다.

## 9. 참고문헌 정책

참고문헌은 계약 조항의 판단 근거처럼 보이게 쓰지 않는다. 실제 확인 가능한 공개 링크를 제공하는 용도다.

초기 참고문헌:

- 소프트웨어사업 계약 및 관리감독에 관한 지침
- 소프트웨어 진흥법
- 한국공정거래조정원 분쟁조정 사례
- 공정위 결정 `18877`, `15923`, `15561`, `1923`
- 대법원 공개문서 `2021다236111`, `2021두48359`

외부 실무 글은 참고문헌으로 직접 싣지 않고, 반복되는 작성 패턴만 제품 문항으로 추상화한다.

추상화한 실무 패턴:

- 요구사항 정의가 없으면 견적과 검수 기준이 흔들린다.
- 결과물 형식과 인도 범위가 없으면 납품 여부가 흔들린다.
- 수정 요청 범위와 횟수 기준이 없으면 무한 수정으로 번진다.
- 유지보수와 운영 지원 범위가 없으면 납품 후 업무가 계약 범위처럼 주장된다.
- 대금 지급 조건과 기성 정산 기준이 없으면 중도 종료 시 분쟁이 커진다.
- 소스코드, 디자인, 콘텐츠, 포트폴리오 사용 가능 범위가 없으면 권리 귀속 분쟁이 생긴다.
- 업무 범위와 제외 범위가 없으면 추가 기능 요구가 기존 계약에 포함된 것처럼 주장된다.

## 10. 웹의 역할

웹은 계약서 편집기가 아니다. 웹은 입력 보조 화면이다.

웹에서 하는 일:

- 역할 선택: 발주자 / 수급자 / 중립
- 계약문서별 입력값 항목 표시
- 문항별 “줄이는 분쟁” 오버레이 메모 표시
- 버튼, 셀렉트, 숫자 입력, 표 입력 제공
- 누락 항목 표시
- 위험 표현 화면 표시
- DOCX 미리보기
- JSON 또는 ZIP 다운로드
- 선택 별첨 포함 여부 결정

웹에서 하지 않는 일:

- 최종 계약 문구 확정
- 법률 판단
- 사용자 대신 계약 체결
- 모든 HWPX/PDF 문서 호환

## 10.1 문서 생성 구조

`scripts/build-docx-package.mjs`는 DOCX 패키징과 ZIP 생성만 담당한다. 각 DOCX의 내용은 `scripts/docx/docs/*.mjs`의 문서별 빌더가 담당한다.

이 구조를 쓰는 이유:

- 문서별 책임이 분리되어 PR 단위 검토가 쉽다.
- 선택 별첨은 해당 빌더가 입력값을 보고 포함 여부를 결정한다.
- 공통 테이블, 문단, 값 포맷 로직은 `scripts/docx/helpers.mjs`에 둔다.

## 10.2 견적서와 로그

견적서의 수량, 단가, 금액은 발주자에게 제출되는 값이다. 제품이 해결해야 할 문제는 단가 공개 여부가 아니라, 해당 단가가 어떤 전제조건, 완수 조건, 테스트 조건, 부수 항목에 묶이는지 계약문서에 반영하는 것이다.

정적 MVP는 입력값을 서버로 보내지 않는다. 웹 서비스로 전환할 경우 견적 row 입력 패턴, 누락 항목, 문항 오버레이 사용 여부 같은 제품 개선 로그를 수집할 수 있다. 이 경우 수집 범위와 익명화 기준은 별도 요구사항으로 둔다.

초기 이벤트 스키마는 `docs/event-schema.md`를 기준으로 한다.

## 11. 품질 기준

MVP 완료 기준:

- 샘플 JSON으로 모든 DOCX가 생성된다.
- 각 DOCX는 Word에서 열 수 있는 패키지 구조를 가진다.
- 계약서 본문에 검수, 하자, 무상수정, 변경요청, 회의, 대금, 권리, 운영비, 개인정보 항목이 들어간다.
- 기능별 완료 기준이 RFP, 검수기준표, 마일스톤 문서에 반영된다.
- 견적 row가 RFP, 계약서, 검수기준표, 대금 지급표, 견적산정표, 변경요청서와 연결된다.
- 견적 row의 전제조건, 완수 조건, 테스트 조건, 부수 항목이 계약문서 표에 자동 반영된다.
- 체크리스트는 내부 점검 룰셋 또는 화면 보조 항목으로만 남긴다.
- 참고문헌 문서에는 실제 확인 가능한 링크가 들어간다.
- 금지 표현이 저장소 주요 문서에 남지 않는다.

## 12. 성공 지표

초기 성공 지표:

- 샘플 입력값으로 DOCX 계약문서 패키지 생성 성공률 100%
- 필수 입력 필드 누락 감지율 90% 이상
- 생성 문서 중 사람이 바로 검토 가능한 문서 비율 80% 이상
- 기능별 완료 기준 입력 누락 감지율 90% 이상
- 계약 검토자가 “추가 확인 필요” 항목을 10분 안에 파악 가능
- 발주자와 수급자 모두 합의표의 균형성을 이해 가능

## 13. 리스크

| 리스크 | 대응 |
|---|---|
| 문서가 너무 많아 사용자가 부담을 느낌 | 기본 패키지와 확장 패키지로 나눈다 |
| 계약서가 한쪽에 치우쳐 보임 | 각 쟁점마다 발주자 이점, 수급자 이점, 균형안을 같이 둔다 |
| 참고문헌이 법률 판단처럼 보임 | 별도 문서로 분리하고 “참고문헌”으로만 표시한다 |
| 웹 UI가 복잡해짐 | 웹은 입력 보조와 누락 표시로 제한한다 |
| DOCX 레이아웃 품질 부족 | 표 폭, 제목, 여백, 서명란을 개선한다 |
| 자유 텍스트가 많아 작성 품질이 흔들림 | 버튼, 셀렉트, 숫자 입력, 표 입력을 기본으로 둔다 |
